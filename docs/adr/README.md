# ADR: решения по архитектуре

ADR — это короткая запись «какое решение приняли и почему».

Так проще через месяц понять, почему проект устроен именно так.

## Как добавлять ADR

1. Создай файл `docs/adr/0002-korotkoe-nazvanie.md`.
2. Заполни блоки: контекст, решение, последствия.
3. Пиши простыми словами, без лишней воды.

## Шаблон

```markdown
# Последняя редакция: DD.MM.YYYY HH:MM UTC+3

# ADR-000X: Название решения

## Контекст

Что было неясно или больно.

## Решение

Что решили сделать.

## Последствия

Что стало лучше и какие есть ограничения.
```

## Что сделать, чтобы применить изменения:

1) `docker compose up -d --build` — Почему: безопасно пересоздать сервисы с учётом изменений инфраструктуры.
2) `docker compose logs -f app nginx queue scheduler` — Почему: проверить ошибки приложения, nginx, очереди и планировщика.

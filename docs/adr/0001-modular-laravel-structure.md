# ADR-0001: Использовать модульную структуру Laravel

## Контекст

В проекте есть несколько каналов и зон ответственности: админка, Telegram, VK, MAX, AI, внешний API и файлы.

Если держать всё в общих `routes/web.php`, `routes/api.php` и случайных сервисах, проект быстро становится трудно читать.

## Решение

Сохраняем модульный подход:

- каждый крупный контур лежит в `app/Modules/<ModuleName>`;
- маршруты подключаются через `*ServiceProvider.php`;
- общая регистрация модулей находится в `bootstrap/providers.php`;
- интерфейс админки строится на Livewire + Blade.

## Последствия

Плюсы:

- проще найти нужную логику;
- новые интеграции можно добавлять отдельным модулем;
- Graphify лучше показывает связи внутри модулей.

Ограничения:

- нужно не забывать регистрировать новый модуль в `bootstrap/providers.php`;
- маршруты распределены по модулям, поэтому для общей картины нужно использовать `php artisan route:list`.

## Что сделать, чтобы применить изменения:

1) `docker compose up -d --build` — Почему: безопасно пересоздать сервисы с учётом изменений инфраструктуры.
2) `docker compose logs -f app nginx queue scheduler` — Почему: проверить ошибки приложения, nginx, очереди и планировщика.

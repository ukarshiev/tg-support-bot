  - `TEST_ADMIN_EMAIL` / `TEST_ADMIN_PASSWORD`.

- Если тестовые доступы не подходят — остановиться и спросить Владыку, не создавать новую учётку.

Plane — основной трекер задач. Linear больше не используется для новых задач и остаётся только историческим источником. Все старые `KAR-*` уже перенесены в Plane и сохранены в описаниях как `Linear ID: KAR-*`.

Plane workspace: `karum`.
Plane URL: `https://plane.karumweb.ru`.
Секреты/API-доступы: Bitwarden Secrets Manager; локальный `services.txt` содержит только Bitwarden-доступ агента.
Отчёты миграции Linear → Plane: `C:\Users\umidt\.codex\migration\`.

Для каждой заметной работы, где требуется запись в changelog, агент обязан в том же рабочем цикле создать или обновить задачу в Plane. Задача в Plane — рабочий артефакт для следующего агента: по ней должно быть понятно, с чего всё началось, что уже сделано, что проверено и что осталось.

Порядок действий:

1. До создания новой задачи найти существующую Plane-задачу по `Linear ID`, названию, файлам, ошибке или области работ.
2. Если подходящая задача уже есть — обновить её описание, комментарий или статус, не создавать дубль.
3. Если подходящей задачи нет — создать новую задачу в Plane в проекте `TgSupportBot`.
4. Если проекта `TgSupportBot` нет в Plane — создать его в workspace `karum`, затем создать задачу.
5. В описании задачи обязательно указать:
   - исходный запрос/проблему Владыки простыми словами;
   - затронутые модули, файлы и сервисы;
   - что было изменено или что планируется изменить;
   - какие проверки/деплой выполнены и с каким результатом;
   - оставшиеся риски, ограничения и следующие шаги;
   - ссылку/упоминание версии из changelog, если запись уже создана.
6. В changelog-записи по возможности указывать Plane-задачу. Если работа продолжает старую мигрированную задачу — дополнительно указывать её `Linear ID: KAR-*`.

Запрещено закрывать работу как полностью готовую, если для changelog-значимого изменения не создана и не обновлена Plane-задача, кроме случая явной недоступности Plane. Если Plane недоступен, это нужно прямо написать в финальном отчёте и не выдумывать номер задачи.

## Graphify

В проект подключён Graphify: локальный граф знаний лежит в `graphify-out/` и помогает быстрее понимать связи между файлами, классами, функциями и маршрутами.

Когда пользователь пишет `/graphify`, сначала используй Graphify, а уже потом обычный поиск по коду.

Правила работы:
- Для вопросов по устройству проекта сначала запускай `graphify query "<вопрос>"`, если есть `graphify-out/graph.json`.
- Для поиска связи между двумя сущностями используй `graphify path "<A>" "<B>"`.
- Для объяснения одной сущности используй `graphify explain "<название>"`.
- Изменения в `graphify-out/` после хуков или обновления графа — это нормально.
- После изменения кода запускай `graphify update .`, чтобы граф не устаревал.

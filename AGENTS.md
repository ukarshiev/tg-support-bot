Обращайся к пользователю как **Владыка**. Отвечай по-русски, коротко и по делу. Если нужно действие от Владыки — объясняй простыми словами: что сделать, зачем и какой будет результат.

## 1. Главный принцип

Главный принцип — **не навреди**.

Агент не должен угадывать состояние проекта и не должен руководствоваться домыслами. Сначала факты, потом выводы и действия.

Если фактов не хватает:

1) прямо скажи Владыке, что именно надо проверить;
2) проверь проект, документацию, логи, Docker, настройки и связанные файлы;
3) только потом предлагай решение или вноси изменения.

Для `tg-support-bot` перед выводами проверяй по ситуации:

- `git status --short` — чтобы не задеть чужой WIP;
- текущую ветку, commit и remote/status — если говоришь про push, deploy или актуальную версию;
- файлы, маршруты, сервисы, миграции, настройки и зависимости;
- Docker-контейнеры и свежие логи — если задача касается runtime;
- официальную документацию и внешние источники — если задача касается библиотек, API, багов или best practice.

## 2. Ошибки: искать корень, а не маскировать симптом

При устранении ошибок агент должен искать первопричину.

Нельзя просто навешивать много заплаток, обходов и избыточных проверок, если можно исправить проблему прямо в корне.

Правильный порядок:

1) воспроизвести или подтвердить проблему фактами;
2) найти участок кода, настройки или интеграции, где возникает причина;
3) проверить официальную документацию компонентов, библиотек и решений;
4) при необходимости проверить bug tracker, release notes, issues и интернет;
5) исправить причину без лишней сложности;
6) проверить, что соседние сценарии не сломались.

## 3. Доработки: сначала оценка, потом реализация

При доработках агент должен анализировать не только просьбу Владыки, но и контекст проекта.

Обязательно:

- оценить плюсы и минусы предлагаемого решения;
- понять, что считается Best Practice для конкретной задачи;
- предложить лучший вариант простыми словами;
- не срезать углы;
- не добавлять избыточность;
- учитывать зависимости `tg-support-bot`, PostEditBot Bridge, AI, Docker и upstream-обновления;
- если предлагается новая библиотека или компонент — проверить, что остальные части системы продолжат работать, либо явно назвать риски.

## 5. SKILLS и сложные задачи

Перед задачей агент должен посмотреть, подходят ли доступные SKILLS.

- Простая/средняя задача — обычный рабочий цикл: проверить → изменить → проверить → отчитаться.
- Сложная задача — использовать `orchestrate`.
- Дополнительных агентов можно подключать для параллельного анализа, сбора фактов и проверок.
- Навыки и дополнительные агенты не должны повышать риск и не должны заменять проверку фактов.

## 6. Что нельзя делать

Запрещено:

- гадать состояние проекта;
- трогать чужой WIP;
- использовать `git add .`;
- коммитить или пушить без явной команды Владыки;
- хранить секреты в коде, README, changelog, документации или логах;
- делать прямой доступ к БД PostEditBot;
- дублировать Toolsy-интеграцию внутри `tg-support-bot`;
- выдумывать подписки, оплаты, статусы или историю доступа, если PostEditBot не вернул профиль;
- менять upstream-файлы без необходимости, если можно держать нашу интеграцию отдельно.

Все runtime-настройки — через БД/UI. `.env` допустим только как bootstrap/fallback.

## 7. Запрещённые формулировки без фактов

Нельзя говорить:

- «запушено» — если не выполнен `git push` и не проверен remote/status;
- «задеплоено» — если Docker-сервисы не пересозданы или не перезапущены;
- «проверено на бою» — если не проверены контейнеры, свежие логи и актуальная версия/commit;
- «CI прошёл» — GitHub Actions отключены, проверка только локальная.

Если проверка была локальная — так и пиши: «проверено локально».

## 8. Upstream-first правило

Этот репозиторий должен обновляться из официального `prog-time/tg-support-bot`.

Правильная модель:

```text
origin   — наш fork, куда пушим свои ветки
upstream — https://github.com/prog-time/tg-support-bot.git
```

Наши изменения держать отдельно:

- `app/Modules/PostEditBotBridge/**` — основная интеграция;
- `docker-compose.relaxa.yml` — локальный Docker override без конфликтов с PostEditBot;
- отдельные миграции/настройки;
- минимальные hooks в upstream-файлах.

Перед обновлением:

1) `git status --short` — проверить WIP.
2) `git fetch upstream` — получить официальный код.
3) `git merge upstream/main` или `git rebase upstream/main` — только если рабочее дерево чистое.
4) Проверить `app/Modules/PostEditBotBridge/**`, AI hook, chat UI и настройки.

## 9. PostEditBot Bridge

Назначение: показать оператору карточку клиента и дать AI безопасный контекст.

Источник правды:

- клиенты;
- подписки;
- платежи;
- статусы;
- история доступа.

Источник — только защищённый endpoint PostEditBot:

```text
GET /api/support/client-profile
Authorization: Bearer <bridge-token>
```

Запрещено:

- подключаться к PostgreSQL PostEditBot напрямую;
- дублировать Toolsy-интеграцию внутри `tg-support-bot`;
- выдумывать подписки/оплаты, если PostEditBot не вернул профиль.

## 10. AI

Режим по умолчанию — `hybrid`.

- `draft` — AI только готовит черновики.
- `hybrid` — AI может отвечать в безопасных случаях.
- `auto` — максимум автоматизации, использовать осторожно.

Для тем оплаты, возвратов, блокировок, восстановления доступа и спорных подписок — эскалация оператору.

AI не должен выдумывать данные клиента. Если PostEditBot Bridge не вернул профиль — AI обязан сказать, что данных нет, и передать оператору.

## 11. Docker

Для локального запуска рядом с PostEditBot использовать:

```bash
docker compose -f docker-compose.yml -f docker-compose.relaxa.yml up -d --build
```

Проверка логов:

```bash
docker compose -f docker-compose.yml -f docker-compose.relaxa.yml logs -f app queue scheduler
```

## 12. Docs/changelog

При изменении кода или правил проекта:

- обновляй документацию;
- обновляй дату последней редакции в начале изменённых `.md`;
- проверяй наличие `changelog.md`;
- если `changelog.md` отсутствует — создай его;
- новую запись добавляй сверху;
- старые записи changelog не редактируй;
- дату новой записи бери по UTC+3;
- если за эту дату уже есть запись — добавляй время `HH:MM`.

Формат записи changelog:

```text
X.Y.Z – DD.MM.YYYY HH:MM
- [Тип изменения] (Область) место — Краткое развернутое описание
```

Формат команды в документации:

```text
Что сделать, чтобы применить изменения:
1) `<команда>` — Почему: `<коротко>`
```

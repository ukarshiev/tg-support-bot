## Боевой режим решения

- Этот solution считается работающим в боевом режиме, даже если запущен локально через Docker.
- Домен `care-support.relaxa.club`, база `tg_support_bot`, реальные пользователи, реальные сообщения и реальные токены — признаки боевой среды.
- Любые действия с БД, очередями, миграциями, smoke/e2e и внешними API выполнять только как production-операции: сначала диагностика, затем дамп, затем явное подтверждение Владыки, затем минимальное изменение.
- Запрещено считать локальный контейнер песочницей без доказательств.
- Если нужно «починить, чтобы solution работал», сначала восстановить доступность сервисов, проверить данные и только потом запускать очереди/миграции.

## Безопасность учётных записей

- Запрещено создавать новые учётные записи без прямой команды Владыки.
- Запрещено менять пароль, email, роль или доступы учётной записи `admin@relaxa.club`.
- Для тестов админки использовать только:
  - `playwright-admin@example.test`;
  - `test@example.com`.
- Пароли не хранить в git. Брать локально из `.env`:
  - `PLAYWRIGHT_ADMIN_EMAIL` / `PLAYWRIGHT_ADMIN_PASSWORD`;
  - `TEST_ADMIN_EMAIL` / `TEST_ADMIN_PASSWORD`.
- Резервная локальная памятка с доступами: `.local-support-credentials.txt`.
- Если тестовые доступы не подходят — остановиться и спросить Владыку, не создавать новую учётку.

## Абсолютная защита баз данных и окружений

Это правило главнее любых задач, планов, тестов, smoke-проверок и желания «быстро проверить».

### Что запрещено без прямого письменного разрешения Владыки

Запрещено выполнять любые действия, которые могут изменить, очистить, пересоздать или мигрировать базу данных:

- `php artisan migrate --force`;
- `php artisan migrate:fresh`;
- `php artisan migrate:refresh`;
- `php artisan migrate:reset`;
- `php artisan db:wipe`;
- `php artisan schema:dump`;
- сиды, фабрики, tinker-скрипты и любые SQL-записи в неявно проверенную БД;
- создание тестовых пользователей, сообщений, токенов, настроек и smoke-данных;
- `docker compose down -v`;
- `docker volume rm`;
- удаление/пересоздание volume PostgreSQL/MySQL/Redis;
- `DROP DATABASE`, `DROP SCHEMA`, `TRUNCATE`, массовые `DELETE`/`UPDATE`;
- любые команды, которые запускают тесты с `RefreshDatabase`, если не доказано, что тесты идут в изолированную sqlite/in-memory БД;
- любые smoke/e2e-проверки, которые пишут данные в БД.

### Обязательная проверка перед любыми командами, связанными с БД

Перед миграциями, тестами, smoke, tinker, сидингом, SQL и Docker-командами агент обязан:

1. Проверить окружение:
   - `APP_ENV`;
   - `APP_URL`;
   - `DB_HOST`;
   - `DB_DATABASE`;
   - `DB_USERNAME`;
   - текущий Docker Compose project;
   - какие volumes подключены к БД.
2. Если `APP_URL` содержит боевой домен, например `care-support.relaxa.club`, или окружение выглядит не как одноразовая песочница — остановиться и спросить Владыку.
3. Проверить, что это не production/staging/общая рабочая БД.
4. Перед любым изменением схемы или данных сделать дамп БД.
5. Написать Владыке простыми словами:
   - к какой БД подключён агент;
   - какая команда будет выполнена;
   - что она может изменить;
   - где лежит свежий дамп;
   - как откатиться.
6. Дождаться прямого подтверждения Владыки.

### Дамп обязателен

Перед любым действием, которое может изменить данные или схему, обязательно создать дамп:

```bash
docker compose exec -T pgdb pg_dump -U <user> -d <database> > backups/<database>-YYYYMMDD-HHMMSS.sql
```

Если дамп не удалось создать — остановиться. Нельзя продолжать «на авось».

### Тесты должны быть изолированы

Перед запуском PHPUnit/Feature/Livewire/e2e:

- проверить `phpunit.xml`;
- проверить `tests/bootstrap.php`;
- убедиться, что `DB_CONNECTION=sqlite` и `DB_DATABASE=:memory:`;
- убедиться, что нет `bootstrap/cache/config.php`, который может подменить настройки;
- если есть сомнения — не запускать тесты и спросить Владыку.

Единственный разрешённый локальный запуск PHPUnit:

```powershell
.\scripts\run-isolated-tests.ps1 [путь-к-тесту]
```

Скрипт отключает сеть контейнера, не подключает Compose volumes, монтирует репозиторий только для чтения и принудительно использует SQLite `:memory:`. Запуск `phpunit`, `php artisan test` или `docker compose run ... app` напрямую запрещён: production-сервис `app` может унаследовать PostgreSQL и общий `bootstrap/cache/config.php`.

### Smoke, Playwright и любые UI/e2e-проверки

Smoke/e2e/UI-проверки считаются потенциально опасными, потому что они могут:

- логиниться в админку;
- создавать пользователей;
- создавать сообщения;
- менять настройки;
- запускать очереди;
- отправлять внешние запросы;
- менять состояние БД через обычный интерфейс.

Запрещено запускать smoke, Playwright, browser-тесты и любые ручные UI-проверки через админку без прямого письменного разрешения Владыки, если заранее не доказано, что среда одноразовая и тестовая.

Перед запуском smoke/e2e агент обязан:

1. Проверить:
   - `APP_ENV`;
   - `APP_URL`;
   - `DB_HOST`;
   - `DB_DATABASE`;
   - `DB_USERNAME`;
   - Docker Compose project;
   - подключённые volumes;
   - что домен не боевой и не похож на боевой.
2. Проверить, что тест идёт в отдельную тестовую БД, а не в рабочую/локально-боевую.
3. Сделать дамп БД до smoke, если smoke может создать или изменить хотя бы одну запись.
4. Написать Владыке:
   - какой URL будет открыт;
   - под каким тестовым пользователем будет вход;
   - какие записи будут созданы/изменены;
   - какие очереди или внешние API могут сработать;
   - где лежит дамп;
   - как откатить изменения.
5. Дождаться прямого подтверждения Владыки.
6. После smoke:
   - показать результат;
   - перечислить созданные/изменённые записи;
   - удалить только свои тестовые данные, если Владыка разрешил;
   - не трогать чужие или реальные данные.

Если smoke нужен только для визуальной проверки, сначала использовать мок/скрин/локальный HTML/изолированную sqlite-среду. Живую админку использовать только последним вариантом и только после подтверждения.

### Нельзя считать локальный Docker безопасным автоматически

Локальный Docker-контейнер может быть подключён к боевой, staging или общей рабочей БД. Сам факт, что команда выполняется на локальном компьютере, не означает безопасность.

Если есть любой признак боевой среды — `APP_URL` с реальным доменом, реальные токены, реальные пользователи, реальные сообщения, нестандартный volume, непонятный `DB_HOST` — агент обязан остановиться.

### Приоритет при сомнениях

Если агент не уверен, безопасна ли БД:

1. Ничего не менять.
2. Ничего не мигрировать.
3. Ничего не сидить.
4. Ничего не очищать.
5. Спросить Владыку.

## Graphify

В проект подключён Graphify: локальный граф знаний лежит в `graphify-out/` и помогает быстрее понимать связи между файлами, классами, функциями и маршрутами.

Когда пользователь пишет `/graphify`, сначала используй Graphify, а уже потом обычный поиск по коду.

Правила работы:
- Для вопросов по устройству проекта сначала запускай `graphify query "<вопрос>"`, если есть `graphify-out/graph.json`.
- Для поиска связи между двумя сущностями используй `graphify path "<A>" "<B>"`.
- Для объяснения одной сущности используй `graphify explain "<название>"`.
- Изменения в `graphify-out/` после хуков или обновления графа — это нормально.
- После изменения кода запускай `graphify update .`, чтобы граф не устаревал.
